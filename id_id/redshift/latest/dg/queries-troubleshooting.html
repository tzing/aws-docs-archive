Koneksi gagal

Koneksi kueri Anda dapat gagal karena alasan berikut; kami menyarankan pendekatan pemecahan masalah berikut.

Klien tidak dapat menyambung ke server

Jika Anda menggunakan sertifikat SSL atau server, pertama-tama hapus kompleksitas ini saat Anda memecahkan masalah koneksi. Kemudian tambahkan sertifikat SSL atau server kembali ketika Anda telah menemukan solusi. Untuk informasi lebih lanjut, pergi keKonfigurasikan Opsi Keamanan untuk Koneksidi dalamPanduan Manajemen Amazon Redshift.

Koneksi ditolak

Umumnya, ketika Anda menerima pesan kesalahan yang menunjukkan bahwa ada kegagalan untuk membuat sambungan, itu berarti ada masalah dengan izin untuk mengakses klaster. Untuk informasi lebih lanjut, pergi keKoneksi ditolak atau gagaldi dalamPanduan Manajemen Amazon Redshift.

Permintaan hang

Kueri Anda dapat menggantung, atau berhenti merespons, karena alasan berikut; kami menyarankan pendekatan pemecahan masalah berikut.

Koneksi ke database dijatuhkan

Kurangi ukuran unit transmisi maksimum (MTU). Ukuran MTU menentukan ukuran maksimum, dalam byte, dari paket yang dapat ditransfer dalam satu bingkai Ethernet melalui koneksi jaringan Anda. Untuk informasi lebih lanjut, pergi keKoneksi ke database dijatuhkandi dalamPanduan Manajemen Amazon Redshift.

Koneksi ke database kali keluar

Koneksi klien Anda ke database tampak hang atau waktu habis saat menjalankan kueri panjang, seperti perintah COPY. Dalam hal ini, Anda mungkin mengamati bahwa konsol Amazon Redshift menampilkan bahwa kueri telah selesai, tetapi alat klien itu sendiri masih tampak menjalankan kueri. Hasil kueri mungkin hilang atau tidak lengkap tergantung kapan koneksi berhenti. Efek ini terjadi ketika koneksi idle diakhiri oleh komponen jaringan menengah. Untuk informasi lebih lanjut, pergi keMasalah Batas Waktu Firewalldi dalamPanduan Manajemen Amazon Redshift.

Sisi klienout-of-memoryterjadi kesalahan dengan ODBC

Jika aplikasi klien Anda menggunakan koneksi ODBC dan kueri Anda membuat set hasil yang terlalu besar untuk muat dalam memori, Anda dapat melakukan streaming hasil yang ditetapkan ke aplikasi klien Anda dengan menggunakan kursor. Untuk informasi selengkapnya, lihat MENYATAKAN dan Pertimbangan kinerja saat menggunakan kursor.

Sisi klienout-of-memoryterjadi kesalahan dengan JDBC

Ketika Anda mencoba untuk mengambil set hasil besar melalui koneksi JDBC, Anda mungkin menemukan sisi klienout-of-memorykesalahan. Untuk informasi selengkapnya, lihat Mengatur JDBC mengambil parameter ukuran.

Ada potensi kebuntuan

Jika ada potensi kebuntuan, coba yang berikut ini:

  • LihatSTV_LOCKSdanSTL_TR_CONFLICTtabel sistem untuk menemukan konflik yang melibatkan pembaruan ke lebih dari satu tabel.

  • GunakanPG_CANCEL_BACKENDberfungsi untuk membatalkan satu atau lebih pertanyaan yang saling bertentangan.

  • GunakanPG_TERMINATE_BACKENDberfungsi untuk mengakhiri sesi, yang memaksa setiap transaksi yang sedang berjalan dalam sesi diakhiri untuk melepaskan semua kunci dan memutar kembali transaksi.

  • Jadwalkan operasi tulis bersamaan dengan hati-hati. Untuk informasi selengkapnya, lihat Mengelola operasi tulis bersamaan.

Query memakan waktu terlalu lama

Permintaan Anda bisa memakan waktu terlalu lama karena alasan berikut; kami menyarankan pendekatan pemecahan masalah berikut.

Tabel tidak dioptimalkan

Atur kunci sortir, gaya distribusi, dan pengkodean kompresi tabel untuk memanfaatkan sepenuhnya pemrosesan paralel. Untuk informasi selengkapnya, lihat Bekerja dengan optimasi tabel otomatis

Query menulis ke disk

Kueri Anda mungkin menulis ke disk untuk setidaknya sebagian dari eksekusi kueri. Untuk informasi selengkapnya, lihat Meningkatkan kinerja kueri.

Kueri harus menunggu kueri lain selesai

Anda mungkin dapat meningkatkan kinerja sistem secara keseluruhan dengan membuat antrian kueri dan menetapkan berbagai jenis kueri ke antrian yang sesuai. Untuk informasi selengkapnya, lihat Menentukan manajemen beban kerja.

Kueri tidak dioptimalkan

Menganalisis menjelaskan rencana untuk menemukan peluang untuk menulis ulang query atau mengoptimalkan database. Untuk informasi selengkapnya, lihat Rencana kueri.

Query membutuhkan lebih banyak memori untuk dijalankan

Jika kueri tertentu membutuhkan lebih banyak memori, Anda dapat meningkatkan memori yang tersedia dengan meningkatkanwlm_query_slot_count.

Database membutuhkan perintah VACUUM untuk dijalankan

Jalankan perintah VACUUM setiap kali Anda menambahkan, menghapus, atau memodifikasi sejumlah besar baris, kecuali Anda memuat data Anda dalam urutan kunci semacam. Perintah VACUUM mengatur ulang data Anda untuk menjaga urutan dan mengembalikan kinerja. Untuk informasi selengkapnya, lihat Tabel penyedot debu.

Sumber daya tambahan untuk memecahkan masalah kueri yang berjalan lama

Berikut ini adalah topik tampilan sistem dan bagian dokumentasi lainnya yang berguna untuk penyetelan kueri:

  • YangSTV_DALAM PENERBANGANtampilan sistem menunjukkan kueri mana yang berjalan di klaster. Hal ini dapat membantu untuk menggunakannya bersama-sama denganSTV_TERBARUuntuk menentukan kueri mana yang sedang berjalan atau baru saja selesai.

  • SYS_QUERY_HISTORYberguna untuk pemecahan masalah. Ini menunjukkan DDL dan DMLquery dengan sifat yang relevan seperti status mereka saat ini, sepertirunningataufailed, waktu yang dibutuhkan masing-masing untuk menjalankan, dan apakah query berjalan pada cluster concurrency-scaling.

  • STL_QUERYTEXTmenangkap teks query untuk perintah SQL. Selain itu,SVV_QUERY_DALAM PENERBANGAN, yang bergabung STL_QUERYTEXT ke STV_INFLIGHT, menunjukkan lebih banyak metadata kueri.

  • Konflik transaksi-kunci dapat menjadi sumber kemungkinan masalah kueri-kinerja. Untuk informasi tentang transaksi yang saat ini menyimpan kunci pada tabel, lihatSVV_TRANSAKSI-TRANSAKSI.

  • Mengidentifikasi kueri yang merupakan kandidat teratas untuk penyetelanmenyediakan kueri pemecahan masalah yang membantu Anda menentukan kueri yang baru-baru ini dijalankan yang paling memakan waktu. Ini dapat membantu Anda memfokuskan upaya Anda pada pertanyaan yang perlu ditingkatkan.

  • Jika Anda ingin menjelajahi manajemen kueri lebih lanjut dan memahami cara mengelola antrian kueri,Menentukan manajemen beban kerjamenunjukkan bagaimana melakukannya. Manajemen beban kerja adalah fitur canggih dan kami merekomendasikan manajemen beban kerja otomatis dalam banyak kasus.

Beban gagal

Pemuatan data Anda dapat gagal karena alasan berikut; kami menyarankan pendekatan pemecahan masalah berikut.

Sumber Data berbedaAWSWilayah

Secara default, bucket Amazon S3 atau tabel Amazon DynamoDB yang ditentukan dalam perintah COPY harus samaAWSWilayah sebagai cluster. Jika data dan klaster Anda berada di Wilayah yang berbeda, Anda menerima kesalahan yang serupa dengan yang berikut ini: 

The bucket you are attempting to access must be addressed using the specified endpoint.

Jika memungkinkan, pastikan klaster Anda dan sumber data Anda berada di Wilayah yang sama. Anda dapat menentukan Wilayah yang berbeda dengan menggunakanREGIONpilihan dengan perintah COPY.

catatan

Jika klaster dan sumber data Anda berbedaAWSWilayah, Anda dikenakan biaya transfer data. Anda juga memiliki latensi yang lebih tinggi.

perintah COPY gagal

Kueri STL_LOAD_ERRORS untuk menemukan kesalahan yang terjadi selama beban tertentu. Untuk informasi selengkapnya, lihat STL_LOAD_ERRORS.

Beban memakan waktu terlalu lama

Operasi beban Anda bisa memakan waktu terlalu lama karena alasan berikut; kami menyarankan pendekatan pemecahan masalah berikut.

COPY memuat data dari satu file

Pisahkan data beban Anda menjadi beberapa file. Saat Anda memuat semua data dari satu file besar, Amazon Redshift dipaksa untuk melakukan pemuatan serial, yang jauh lebih lambat. Jumlah file harus kelipatan dari jumlah irisan di cluster Anda, dan file harus berukuran hampir sama, antara 1 MB dan 1 GB setelah kompresi. Untuk informasi selengkapnya, lihat Praktik terbaik Amazon Redshift untuk mendesain kueri.

Operasi beban menggunakan beberapa perintah COPY

Jika Anda menggunakan beberapa perintah COPY bersamaan untuk memuat satu tabel dari beberapa file, Amazon Redshift dipaksa untuk melakukan beban serial, yang jauh lebih lambat. Dalam hal ini, gunakan perintah COPY tunggal.

Memuat data tidak benar

Operasi COPY Anda dapat memuat data yang salah dengan cara berikut; kami menyarankan pendekatan pemecahan masalah berikut.

File yang salah dimuat

Menggunakan awalan objek untuk menentukan file data dapat menyebabkan file yang tidak diinginkan untuk dibaca. Sebagai gantinya, gunakan file manifes untuk menentukan file mana yang akan dimuat dengan tepat. Untuk informasi lebih lanjut, lihatcopy_from_s3_manifest_filepilihan untuk perintah COPY danExample: COPY from Amazon S3 using a manifestdalam contoh COPY.

Mengatur JDBC mengambil parameter ukuran

Secara default, driver JDBC mengumpulkan semua hasil untuk query pada satu waktu. Akibatnya, ketika Anda mencoba untuk mengambil hasil besar set melalui koneksi JDBC, Anda mungkin menemukan sisi klienout-of-memorykesalahan. Untuk memungkinkan klien Anda mengambil set hasil dalam batch, bukan dalam satuall-or-nothingfetch, mengatur JDBC mengambil parameter ukuran dalam aplikasi klien Anda.

catatan

Ukuran pengambilan tidak didukung untuk ODBC.

Untuk kinerja terbaik, atur ukuran pengambilan ke nilai tertinggi yang tidak menyebabkan kesalahan memori kehabisan. Nilai ukuran pengambilan yang lebih rendah menghasilkan lebih banyak perjalanan server, yang memperpanjang waktu eksekusi. Server cadangan sumber daya, termasuk slot query WLM dan memori terkait, sampai klien mengambil seluruh hasil set atau query dibatalkan. Bila Anda menyetel ukuran pengambilan dengan tepat, sumber daya tersebut dirilis lebih cepat, membuatnya tersedia untuk kueri lain.

catatan

Jika Anda perlu mengekstrak kumpulan data besar, sebaiknya gunakanMEMBONGKARpernyataan untuk mentransfer data ke Amazon S3. Saat Anda menggunakan UNLOAD, node komputasi bekerja secara paralel untuk mempercepat transfer data.

Untuk informasi lebih lanjut tentang pengaturan JDBC mengambil ukuran parameter, pergi keMendapatkan hasil berdasarkan kursordalam dokumentasi PostgreSQL.
